# Vision2PointFilter 算法技术方案

## 1. 项目目标

Vision2PointFilter 面向自动驾驶离线数据处理场景，核心目标是对车辆运动过程中的 LiDAR 单帧点云进行运动补偿，并将补偿后的点云投影到前视广角图像上，用于验证点云、轨迹、标定和图像之间的几何一致性。

当前项目当前已经具备三项核心能力：

- 点云运动补偿：消除单帧 LiDAR 扫描期间车辆运动造成的点云拖影和形变。
- 点云图像投影：将补偿后的点云投影到配置相机图像，生成几何对齐检查图。
- 语义标签回填：基于配置语义图，为导出点云补充单一 `semantic_label` 字段。

该项目是后续语义点云生成、多相机融合和点云过滤的基础处理模块。

## 2. 总体算法流程

整体处理链路如下：

```text
轨迹解析 -> 位姿插值 -> 点云运动补偿 -> 多相机语义匹配 -> 相机投影 -> PCD/PNG 输出
```

主要步骤：

1. 解析组合导航轨迹，得到车辆在全局坐标系下的位置和姿态。
2. 根据点云帧时间和点内时间偏移，为每个点计算真实采样时刻。
3. 对每个点的采样时刻进行车辆位姿插值。
4. 将点从采样时刻的 LiDAR 坐标系统一变换到当前帧参考坐标系，完成运动补偿。
5. 按配置相机顺序为每个点执行语义图查询，得到单一 `semantic_label`。
6. 将补偿后的点云转换到相机坐标系，并使用相机内参投影到图像平面。
7. 将投影点叠加到各相机语义图上，输出可视化结果和带语义标签的 PCD。

该流程先完成点云去畸变，再进行图像投影，避免扫描畸变影响几何验证。

## 3. 点云运动补偿算法

### 3.1 问题背景

旋转式 LiDAR 的一帧点云不是同一时刻采集完成的。车辆在扫描周期内持续运动，因此帧内不同点对应不同的车辆位姿。如果直接把整帧点云视为同一时刻采集，会导致路沿弯曲、杆状物倾斜、点云拖影等问题。

运动补偿的目标是：把同一帧内所有点统一到该帧参考时刻的 LiDAR 坐标系下。

### 3.2 位姿插值

组合导航轨迹提供离散时刻的车辆位置和姿态。算法将每个轨迹样本表示为：

```text
Pose(t) = { position(t), orientation(t) }
```

对任意点的采样时刻，在相邻轨迹样本之间进行插值：

- 平移采用线性插值。
- 旋转采用四元数球面线性插值。

相比直接插值欧拉角，四元数插值能更稳定地描述连续旋转，避免姿态跳变和奇异性问题。

### 3.3 补偿模型

设当前点原始坐标为 `p_lidar_raw`，点采样时刻的 LiDAR 位姿为 `T_world_lidar_point`，当前帧参考时刻的 LiDAR 位姿为 `T_world_lidar_frame`。

运动补偿过程为：

```text
p_world = T_world_lidar_point * p_lidar_raw
p_lidar_deskew = inverse(T_world_lidar_frame) * p_world
```

含义是：

1. 先将点从自身采样时刻的 LiDAR 坐标系转换到全局坐标系。
2. 再将全局坐标系下的点转换回当前帧参考时刻的 LiDAR 坐标系。

完成后，同一帧内所有点都处于统一时刻的坐标表达中，点云结构更加一致。

### 3.4 质量控制

运动补偿阶段设置了基础过滤策略：

- 轨迹插值间隔过大时，认为该点位姿不可靠并跳过。
- 超过最大处理距离的点不参与输出。
- 无法完成位姿插值的点不参与输出。

这些策略优先保证输出点云的几何可靠性。

## 4. 多相机投影与语义回填算法

### 4.1 投影目的

投影阶段既用于几何验证，也承担语义回填。通过将补偿后的 LiDAR 点云绘制到配置相机图像上，可以直观观察点云是否与道路、车辆、路沿、杆状物等图像结构对齐，并将图像语义回写到导出点云。

投影结果可以帮助发现：

- LiDAR 与相机外参偏差。
- 相机内参或图像尺寸不匹配。
- 点云与图像时间同步异常。
- 点云运动补偿效果不佳。

### 4.2 配置协议

当前 `deskew_clip_export.projection` 使用显式多相机配置：

```yaml
projection:
  enabled: true
  image_root_subdir: images_seg_mask2former
  image_model: undistorted
  camera_names: [back, front_wide, left_back, left_front, right_back, right_front]
  output_subdir: projection
  max_time_diff_ms: 100.0
  point_radius_px: 2
  intensity_color_map: turbo
```

关键约束如下：

- `camera_names` 显式定义参与投影和语义融合的相机集合。
- `camera_names` 的顺序同时定义语义融合顺序和 PNG 输出处理顺序。
- `image_model` 显式定义输入图像的像素模型：
  - `undistorted` 表示输入目录中的图像已完成去畸变。
  - `raw` 表示输入目录中的图像仍为原始带畸变图。
- 旧字段 `projection.image_subdir` 已废弃，继续使用会直接报错。
- 不再保留 `camera_names.front() == front_wide` 的旧约束，首相机可由配置决定。

### 4.3 时间匹配

每帧点云会在每一路相机图像序列中查找时间戳最近的图像：

```text
image* = argmin | timestamp_image - timestamp_lidar |
```

若任一路最近图像与点云帧的时间差超过阈值，则停止处理并报错，避免生成具有误导性的投影和语义结果。

### 4.4 投影模型

补偿后的点先从 LiDAR 坐标系变换到车体坐标系，再变换到相机坐标系：

```text
p_car = T_car_lidar * p_lidar_deskew
p_cam = T_cam_car * p_car
```

随后使用针孔相机模型投影到图像平面：

```text
u = fx * x / z + cx
v = fy * y / z + cy
```

投影阶段会过滤相机后方点、非法点和图像范围外点，只保留有效投影结果。

当 `image_model=raw` 时，像素投影不再使用简单针孔模型，而是基于相机标定中的畸变参数决定原图投影路径：

- 若畸变系数的有效非零项不超过 4 个，则按 4 参数 fisheye 模型投影。
- 若有效非零项超过 4 个，则按 8 参数畸变模型投影。

当 `image_model=undistorted` 时，继续使用针孔投影路径。

因此，当前系统可以同时兼容：

- 去畸变语义图输入
- 原始带畸变语义图输入

且两者都共用同一套多相机时间匹配与语义回填框架。

### 4.5 语义回填策略

当前语义回填采用“按配置顺序命中首个有效标签”的简单融合规则：

1. 对单个点按 `camera_names` 顺序依次投影到各路语义图。
2. 若当前相机投影失败、越界或灰度值没有映射语义标签，则继续下一路。
3. 命中第一路有效语义标签后立即返回，作为该点最终 `semantic_label`。
4. 若全部相机都未命中，则该点 `semantic_label = -1`。

该策略的特点是：

- 只保留一个 `semantic_label` 字段，不为每个相机分别落字段。
- 不做投票、平均、置信度融合和遮挡推理。
- 相机顺序具有明确业务语义，因此必须由配置显式控制。

### 4.6 可视化与输出策略

有效投影点根据 LiDAR 强度值进行伪彩色渲染，并叠加到对应相机语义图上。

当前输出链路进一步区分了“输入图像类型”和“最终 PNG 期望形态”：

- 当 `image_model=undistorted` 时：
  - 直接在去畸变图坐标系中完成渲染。
  - 最终输出 PNG 与当前渲染结果一致。
- 当 `image_model=raw` 时：
  - 仍先在原图坐标系中完成点投影与语义图叠加。
  - 语义取值逻辑保持不变，不额外切换为去畸变图坐标。
  - 仅在最终保存 PNG 前，对“底图 + 投影点”的渲染结果图执行一次去畸变。
  - 最终输出的是去畸变后的投影图。

这样设计的目的在于：

- 保持原有语义回填与原图投影逻辑不变，减少主流程侵入。
- 使 raw 输入模式下导出的投影图仍然方便人工按去畸变视角检查几何对齐效果。

输出协议如下：

- 语义点云输出：
  - `<output_dir>/deskew_pcd/<timestamp>.pcd`
- 投影 PNG 输出：
  - `<output_dir>/projection/<camera_name>/<timestamp>.png`

其中每个相机单独输出一个目录，避免不同视角结果互相覆盖。

这种方式的优点是：

- 保留点云反射强度信息，便于观察路面和高反物体。
- 语义融合与可视化共用同一组时间匹配和投影逻辑。
- 输出结果直观，便于人工快速判断标定和同步质量。
- raw 输入模式下仍能输出便于检查的去畸变投影图，同时不破坏原有语义回填路径。

### 4.7 原图输入去畸变输出实现补充

为支持 `raw` 输入但输出去畸变投影图，投影模块新增了独立的渲染结果图去畸变能力。

实现原则如下：

1. `RenderProjection(...)` 只负责在输入图像坐标系中绘制投影点。
2. 若 `image_model=raw`，则在应用层导出阶段调用渲染结果图去畸变函数。
3. 若 `image_model=undistorted`，则保持原有直接写盘逻辑。

图像去畸变分支同样按相机畸变模型区分：

- `kFisheye4` 使用 fisheye 去畸变路径。
- `kDistorted8` 使用普通畸变模型去畸变路径。

同时在数值稳定性上增加了保护：

- 当 8 参数畸变模型的径向分母接近 0 时，当前帧导出会直接报错退出，避免输出错误重采样结果。

该实现将“语义与投影几何逻辑”和“最终可视化导出形态”解耦，适合离线质检场景。

## 5. 方案优势

当前方案具有以下特点：

- 算法链路清晰：先运动补偿，再图像投影，符合几何处理逻辑。
- 位姿表达稳定：平移线性插值结合四元数旋转插值，适合车辆连续运动场景。
- 坐标变换完整：以全局坐标作为中间表达，实现帧内不同采样时刻点的统一。
- 多相机扩展明确：相机集合、融合顺序和输出目录全部由配置显式控制。
- 原图兼容性更强：同一套流程可以直接接收带畸变语义图输入。
- 验证方式直观：投影图可以直接暴露外参、内参和时间同步问题。
- 导出结果更适合人工检查：raw 输入模式下也能输出去畸变投影视图。
- 扩展基础良好：后续可继续扩展更复杂的语义融合和点云过滤。

## 6. 标定与全局坐标系实现细节

### 6.1 前视广角相机标定读取

当前前视广角投影模块只读取业务实际需要的相机内参和外参，不再依赖去畸变专用 key。

具体读取项为：

- 相机内参：`camera-front-wide.param.cam_matrix.data`
- 图像宽高：`camera-front-wide.param.width`、`camera-front-wide.param.height`
- 相机到车体外参：`camera-front-wide-to-car.param.sensor_calib.data`

实现策略是：

- 只对上述内外参和图像尺寸做加载与合法性检查。
- 畸变参数、`camera-front-wide-undistort`、`camera-front-wide-to-car-undistort` 等无关字段全部忽略。

这样做可以避免投影链路继续绑定去畸变标定格式，使相机标定来源更加单一明确。

### 6.2 全局坐标系模式

为适配不同数据处理需求，运动补偿增加了显式全局坐标系配置：

```yaml
pose:
  coord_frame: enu | utm
```

两种模式共用同一份 GNSS 到 LiDAR 外参文件：

- `calib_extract/calib_gnss_to_lidar_top_ENU.json`

也就是说，`enu` 和 `utm` 的差别不在外参文件选择，而在轨迹坐标解释与姿态处理逻辑。

### 6.3 UTM 模式

在 `utm` 模式下：

- 组合导航轨迹位置直接使用 UTM 坐标。
- yaw 保留 UTM convergence 补偿。
- 运动补偿中的全局位姿 `T_world_lidar` 以 UTM 世界坐标系构造。

该模式适合直接在投影和补偿过程中保持绝对 UTM 地图坐标语义。

### 6.4 ENU 模式

在 `enu` 模式下，行为与 `icv_mapping` 对齐：

- 关闭 UTM convergence 补偿。
- 以首个 LiDAR 帧时间戳之前的轨迹样本作为局部原点参考。
- 将全部轨迹样本从经纬高转换为局部 ENU 坐标。
- 再对样本做一次原点重心化，使轨迹数值集中在局部坐标附近。

因此在实现层面，轨迹字段仍沿用同一个位置容器，但其数值语义在 `enu` 模式下变为局部 ENU，而不是 UTM。

### 6.5 运动补偿统一框架

尽管支持两种全局坐标系，点云运动补偿主算法本身保持统一：

1. 为点时刻插值全局位姿。
2. 将点从采样时刻 LiDAR 坐标系变换到全局坐标系。
3. 再变换回当前帧参考时刻的 LiDAR 坐标系。

统一框架的好处是：

- `enu` / `utm` 模式只在轨迹预处理和姿态解释上分叉。
- 补偿核心公式不需要维护两套不同实现。
- 更容易做回归测试和结果对比。

## 9. 当前能力边界

当前项目主要服务于离线几何处理和质量检查，仍有以下边界：

- 只做单帧点云运动补偿，不做多帧融合建图。
- 虽然已经支持语义回填，但仍只保留单一 `semantic_label`，不输出每相机独立标签。
- 当前投影未加入 z-buffer，暂不处理不同深度点之间的遮挡关系。
- 当前多相机融合仅支持“按顺序命中首个有效标签”，未实现投票、置信度或遮挡推理。
- 最终效果依赖轨迹质量、标定精度和传感器时间同步质量。

## 10. 后续演进方向

后续建议按以下方向推进：

1. 将真实数据集运行纳入半自动或自动化回归，形成稳定的端到端验证链路。
2. 在投影阶段增加深度排序或 z-buffer，提高可视化准确性。
3. 为语义映射文件增加显式配置项，去掉当前对配置文件相对路径的隐式依赖。
4. 扩展多相机融合策略，根据视角、距离和遮挡关系选择最佳图像来源。
5. 将输出点云接入后续过滤、融合或建图流程。

## 11. 汇报结论

Vision2PointFilter 当前已经建立了一条完整的 LiDAR 离线预处理链路：通过组合导航轨迹对单帧点云进行逐点运动补偿，再基于显式配置的多相机集合执行语义查询与投影输出，最终生成带 `semantic_label` 的 PCD 和逐相机 PNG。该方案已经在真实数据集上完成 201 帧全量验证，可作为后续语义点云过滤和多相机点云融合的基础模块。
